# 数据跨境合规审核法律智能体技术方案

版本：V0.1  
日期：2026-05-31  
范围：MVP / 试点版

## 1. 目标与边界

本方案面向“数据跨境合规审核法律智能体”MVP 试点版，目标是建设一个可完成企业信息采集、法律知识库检索、初步风险识别、问题清单生成和 PDF 报告输出的系统。

系统定位为“初步合规筛查工具”，不出具正式法律意见，不替代律师、合规顾问或监管机关判断。

MVP 支持范围：

1. 国家/地区：中国、越南、新加坡。
2. 场景：境内数据合规初步审查、数据出海/海外落地初步审查。
3. 用户规模：30 家以内企业试点。
4. 输出：页面版审查结果、PDF 体检报告、结构化问题清单。

MVP 暂不覆盖：

1. 正式法律意见书。
2. 监管申报、备案、合同签署等法律手续。
3. 并购、复杂跨国集团、金融牌照、医疗数据等高复杂场景。
4. 全国家/地区覆盖。

## 2. 总体架构

系统建议拆分为以下模块：

1. 企业端前端：企业注册登录、任务创建、隐私承诺确认、5 轮问答、报告查看与下载。
2. 管理后台前端：企业管理、任务管理、法律资料管理、标签管理、报告模板管理。
3. 后端 API 服务：用户、企业、任务、问答、报告、权限、日志等业务接口。
4. 智能体服务：问答编排、槽位抽取、风险分析、问题清单生成、报告内容生成。
5. 法律知识库服务：法律资料入库、正文清洗、条文切片、标签检索、向量/全文检索、依据溯源。
6. PDF 报告服务：HTML 报告渲染、PDF 生成、文件存储。
7. PolarDB：业务数据、法律资料元数据、条文切片、问答记录、风险项、报告记录。
8. 文件存储：原始法律资料、清洗后文本、PDF 报告、导入批次文件。
9. 日志与监控：操作日志、模型调用日志、检索日志、报告生成日志。

建议 MVP 优先采用“模块化单体 + 独立知识库处理任务”的方式实现，避免过早拆成多个微服务。后续业务稳定后，再将智能体服务、知识库检索服务、PDF 服务独立部署。

## 3. 推荐技术路线

### 3.1 前端

建议采用：

1. React / Next.js。
2. 企业端与管理后台共用组件体系。
3. 页面路由按业务域拆分：认证、任务、问答、报告、后台管理。

关键页面：

1. 登录/注册页。
2. 企业工作台。
3. 新建审查任务页。
4. 隐私承诺确认页。
5. 智能体问答页。
6. 报告详情页。
7. 报告历史页。
8. 后台企业列表页。
9. 后台任务详情页。
10. 法律资料管理页。
11. 标签管理页。

### 3.2 后端

建议采用：

1. Python FastAPI 或 Node.js NestJS。
2. REST API 作为 MVP 主接口。
3. 后台任务队列处理报告生成、知识库清洗、向量生成等异步任务。
4. JWT / Session 均可，MVP 优先简单可靠。

核心业务域：

1. Auth：登录、注册、角色权限。
2. Company：企业信息。
3. AuditTask：审查任务。
4. Conversation：5 轮问答记录。
5. SlotExtraction：结构化字段抽取。
6. KnowledgeBase：法律资料、条文切片、检索。
7. RiskAnalysis：风险项和问题清单。
8. Report：页面报告和 PDF 报告。
9. Admin：后台管理。
10. AuditLog：操作日志。

### 3.3 数据库

### 3.4 智能体与模型

智能体采用“固定流程编排 + 模型能力辅助”的方式，而不是完全开放式聊天。

MVP 智能体能力拆成 5 个节点：

1. Round Planner：根据任务类型生成当前轮问题。
2. Slot Extractor：从用户回答中抽取结构化字段。
3. Missing Field Checker：判断缺失字段和追问项。
4. Retrieval Planner：根据任务国家、数据类型、跨境情况生成检索条件。
5. Risk Reporter：结合检索结果生成风险项、问题清单和报告草稿。

关键约束：

1. 所有法律依据必须来自知识库检索结果。
2. 未检索到依据时，输出“依据不足”或“建议人工复核”。
3. 不输出“完全合规”“必然违法”等绝对结论。
4. 风险判断使用“初步判断”“可能存在”“建议进一步确认”等表达。
5. 高风险、重要数据、敏感个人信息、依据不足事项必须进入人工复核提示。

## 4. 核心业务流程

### 4.1 企业审查流程

1. 企业用户注册或登录。
2. 企业创建审查任务。
3. 用户选择审查类型：境内合规审查 / 数据出海审查。
4. 用户选择国家：仅中国，或中国 + 越南 / 新加坡。
5. 用户确认隐私承诺和使用限制。
6. 系统开启 5 轮问答。
7. 每轮用户回答后，系统保存消息并抽取槽位。
8. 第 5 轮结束后，系统检查缺失信息并补充追问。
9. 系统根据国家、数据类型、数据流向检索法律知识库。
10. 系统生成风险项、问题清单、整改建议。
11. 系统生成页面报告。
12. 系统异步生成 PDF。
13. 用户查看和下载报告。

### 4.2 后台资料入库流程

1. 管理员上传法律资料或导入数据包。
2. 系统记录导入批次。
3. 系统解析元数据：国家、法规名称、发布机关、生效日期、来源 URL、标签等。
4. 系统清洗正文 HTML，输出纯文本。
5. 系统按国家规则做条文切片。
6. 系统给切片挂载标签。
7. 系统生成摘要和检索字段。
8. 系统建立全文索引或向量索引。
9. 管理员抽检并确认有效。

### 4.3 报告生成流程

1. 汇总企业基础信息和审查范围。
2. 汇总结构化槽位。
3. 汇总检索到的法律依据。
4. 生成风险项。
5. 生成问题清单。
6. 生成整改建议。
7. 插入免责声明和使用限制。
8. 渲染 HTML 报告。
9. 转换为 PDF。
10. 保存报告记录和文件地址。

## 5. 知识库设计

### 5.1 法律资料分层

建议将法律资料拆成四层：

1. Source File：原始文件或网页快照。
2. Legal Document：法规级元数据。
3. Legal Clause / Chunk：条文级或段落级切片。
4. Retrieval Index：全文索引、标签索引、向量索引。

### 5.2 切片策略

不同国家采用不同切片规则：

1. 中国：优先按“第 X 条”切片，保留章、节上下文。
2. 新加坡：优先按 Part、Section、Regulation 切片。
3. 越南：优先按 “Điều” 切片，保留 Chapter / Section 上下文。

每个切片至少包含：

1. document_id
2. country
3. clause_no
4. clause_title
5. clause_content
6. clause_summary
7. tags
8. source_url
9. effective_status
10. version

### 5.3 检索策略

MVP 建议采用混合检索：

1. 国家过滤：只在任务相关国家内检索。
2. 领域过滤：优先 cross_border_data，其次 enterprise_landing，tax 暂不作为核心依据。
3. 标签过滤：个人信息、敏感个人信息、数据出境、跨境传输、数据本地化、告知同意、安全措施等。
4. 关键词/全文检索：保证可解释性。
5. 向量检索：提升自然语言召回。
6. 重排序：按国家匹配、标签匹配、优先级、有效状态、语义相似度排序。

报告引用时必须保留：

1. 法规名称。
2. 条文编号或切片位置。
3. 摘要。
4. 来源 URL。
5. 适用原因。

## 6. 风险判断设计

风险判断采用“规则兜底 + 模型归纳”的方式。

### 6.1 初始规则

1. 涉及个人信息跨境传输：至少标记为中风险或需重点审查。
2. 涉及敏感个人信息或敏感数据：提高一个风险等级。
3. 涉及重要数据或监管敏感行业：标记高风险或人工复核。
4. 境外接收方不明确：进入问题清单。
5. 用户告知或授权不充分：进入风险项。
6. 数据处理协议缺失：进入风险项或整改建议。
7. 安全措施不明确：进入问题清单和整改建议。
8. 法律依据不足：标记“信息不足，需补充判断”。

### 6.2 风险等级

1. 低风险：信息较完整，未触发重点规则，但仍需保留审查提示。
2. 中风险：涉及个人信息、跨境流动、授权/协议/安全措施存在不确定。
3. 高风险：涉及敏感个人信息、重要数据、监管敏感行业、接收方不明、依据冲突或严重缺失。
4. 信息不足：关键字段缺失，无法形成稳定判断。

## 8. 安全与权限

### 8.1 角色

1. enterprise_user：企业普通用户。
2. enterprise_admin：企业管理员。
3. platform_admin：平台管理员。
4. legal_admin：法律资料管理员。

### 8.2 权限原则

1. 企业用户只能访问所属 company_id 下的数据。
2. 报告下载必须校验用户、企业和任务归属。
3. 后台接口必须校验管理员角色。
4. 法律资料管理仅 legal_admin 和 platform_admin 可操作。
5. 所有关键操作写入 operation_logs。

### 8.3 数据保护

1. 密码使用强哈希算法存储。
2. 联系方式、报告下载地址等敏感字段脱敏展示。
3. PDF 文件使用带权限校验的下载接口，不直接暴露永久公开地址。
4. 模型调用日志避免记录不必要敏感信息。
5. 支持企业数据保留策略和删除策略的后续扩展。

## 9. 部署建议

MVP 部署可采用：

1. Web 前端：静态部署或 Node 服务部署。
2. API 后端：单实例起步，支持水平扩展。
3. 后台任务：独立 worker。
4. PolarDB：主业务库。
5. 文件存储：对象存储。
6. 向量检索：独立服务或数据库扩展，按后续数据量决定。

环境建议：

1. dev：本地开发。
2. test：联调和验收。
3. pilot：试点企业使用。
4. prod：正式商用后启用。

## 10. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 法律资料仍是 HTML 原文 | 检索噪音大，报告依据不稳 | 先做正文清洗、条文切片、人工抽检 |
| 模型编造法律依据 | 法律风险高 | 报告依据必须绑定知识库来源 |
| 用户回答信息不足 | 风险判断不可靠 | 设置必填槽位、缺失字段追问、信息不足标识 |
| 国家法规差异大 | 切片和标签难统一 | 按国家配置切片规则和标签映射 |
| PDF 报告格式不稳定 | 影响交付观感 | 先固定 HTML 模板，再生成 PDF |
| 企业数据串读 | 严重安全问题 | company_id 数据隔离、权限校验、测试覆盖 |

## 11. MVP 验收口径

1. 企业可以注册、登录、创建审查任务。
2. 企业可以选择中国、越南、新加坡相关审查范围。
3. 系统可以完成 5 轮问答并保存记录。
4. 系统可以抽取核心结构化字段。
5. 系统可以按国家调用知识库。
6. 系统可以生成可溯源的风险项。
7. 系统可以生成问题清单和整改建议。
8. 系统可以生成页面报告和 PDF 报告。
9. 企业只能访问本企业数据。
10. 管理后台可以维护法律资料和查看任务。

## 12. 下一步接数据时的处理顺序

用户提供法律数据包后，优先执行：

1. 检查目录结构和文件编码。
2. 过滤 __MACOSX、.DS_Store、._* 等无效文件。
3. 解析主数据 JSON / CSV / NDJSON。
4. 重写不可用的本机绝对路径。
5. 清洗 content.txt 中的 HTML。
6. 输出 clean_text。
7. 按国家切条文。
8. 生成 legal_documents 和 legal_clauses 导入数据。
9. 生成标签和检索索引。
10. 用样例问题验证召回质量。
